## The bug (now fixed)
Comtrade getFinalData with customsCode=None and motCode=None returns each
annual export value decomposed along two dimensions:
- customsCode: C00 (all procedures), C03/C06/C07 (specific)
- motCode: 0 (all modes), 9110 pipeline, 3100 sea, 1000 air, etc.

For AZE 2019-2024 this gave ~1,148 rows per (reporter, year). Old code used
drop_duplicates(keep='first') which is order-dependent on the API response.
For some country-years it picked sub-aggregate rows instead of the
C00/motCode=0 row, producing tiny denominators and 100% resource shares.

Fix (in both pull_one and pull_total):
1. Pass customsCode='C00' and motCode='0' to API (filter at source)
2. Defensive client-side filter on those same values
3. groupby([...])['primaryValue'].max() instead of drop_duplicates
4. Uniform 6-year batches (12-year TOTAL batches timed out)
5. period dtype coerced to int at every cache boundary

Smoke-tested with synthetic data: filter went 11 -> 6 -> 2 correctly,
picking AZE 19.64B and NOR 102.94B as expected.

## Key output metrics produced by e1b
- tight_share = (HS26 + HS27 + HS71) / TOTAL
- extractives_share = sum of (25, 26, 27, 44, 71) / TOTAL
- base_metals_share = sum of (72, 74, 75, 76, 78, 79, 80, 81) / TOTAL
- wide_resource_share = extractives + base_metals + (28+29) / TOTAL
  (HEADLINE; replaces "Total natural resources rents (% of GDP)")
- hydrocarbon_share = HS27 / TOTAL (replaces oil + gas rents)
- ores_share = HS26 / TOTAL (replaces mineral rents)
- precious_share = HS71 / TOTAL
- resource_herfindahl = concentration of resource basket
- Per-chapter shares: hs25_share through hs81_share
